# 代码规范

## 1. 文件体积与拆分（硬性）

- **单文件目标 300 行以内，上限 500 行**。超过 500 行必须拆分，不接受例外。
- 行数统计含模板/样式/脚本全部内容（Vue SFC 按整文件计）。
- 拆分方向（按优先级）：
  1. **按职责拆**：一个文件只做一件事。组件过大 → 拆子组件；Service 过大 → 按业务域拆多个 Service。
  2. **抽逻辑**：前端把复杂逻辑抽到 `composables/`（`useXxx`）或 `utils/`；后端把可复用逻辑下沉到 `service` / `common`。
  3. **抽配置/常量/类型**：长的常量表、类型定义、表格列定义独立成 `xxx.config.ts` / `types.ts` / `columns.ts`。
- **函数单体 ≤ 80 行、圈复杂度尽量 ≤ 10**；超出请拆小函数。
- 嵌套层级 ≤ 4 层，深层嵌套用**卫语句（early return）** 拍平。

## 2. 前端：Vue SFC 结构

统一顺序：`<script setup lang="ts">` → `<template>` → `<style scoped>`。

```vue
<script setup lang="ts">
// 1) import  2) defineOptions/defineProps/defineEmits  3) 响应式状态
// 4) computed  5) 方法  6) 生命周期/watch
defineOptions({ name: "OrderList" });
</script>

<template>
  <!-- 模板只做展示与事件绑定，复杂逻辑放 script -->
</template>

<style scoped lang="scss"></style>
```

- 一律 **`<script setup>` + Composition API**，禁止 Options API（与 pure-admin/Vant 生态一致）。
- 每个组件必须 `defineOptions({ name })`，名称用 **PascalCase**，便于 keep-alive 与调试。
- `<style>` 默认 `scoped`；需要穿透用 `:deep()`，禁止全局裸样式污染。
- 模板内禁止写复杂表达式/三元嵌套，抽成 `computed` 或方法。
- Props 必须声明类型与默认值；事件用 `defineEmits` 显式声明。

## 3. 前端：逻辑分层

- **视图** `views/`：页面级，组合组件与 store，不写可复用通用逻辑。
- **组件** `components/`：纯展示/交互复用单元，通过 props/emit 通信，不直接依赖具体业务 store。
- **组合式函数** `composables/`：跨组件复用的有状态逻辑，命名 `useXxx`。
- **接口** `api/`：只负责请求定义与出入参类型，不含 UI/业务判断。
- **状态** `store/`：见[状态管理](#7-前端状态管理pinia)。
- **工具** `utils/`：纯函数，无副作用，无框架依赖。

## 4. 后端：Java 分层（严格单向依赖）

`controller → service → mapper`，**禁止跨层/反向调用**（controller 不直接调 mapper，entity 不进 controller 出参）。

| 包 | 职责 | 红线 |
| --- | --- | --- |
| `controller` | 接收请求、参数校验、组装 `Result` | 不写业务逻辑、不碰事务 |
| `service` | 业务逻辑、事务边界（`@Transactional`） | 入参用 DTO，不接收 `HttpServletRequest` |
| `mapper` | 持久化（MyBatis/JPA） | 只做数据存取，无业务判断 |
| `entity` | 数据库映射对象 | **禁止**作为接口出入参直接暴露 |
| `dto` | 接口出入参 | 与 entity 分离，字段按接口需要裁剪 |
| `common` | `Result` / 异常 / 通用工具 | 不依赖具体业务 |
| `config` | 配置类（CORS、拦截器等） | — |

- 一个 `Controller` 聚焦一个资源；方法对应一个端点，体积超标按业务域拆多个 Controller。
- 事务只加在 `service` 方法上，注意 `@Transactional` 自调用失效问题。

## 5. 注释规范

- 注释解释 **“为什么”**，不复述 **“做了什么”**；代码即文档，命名清晰优先于堆注释。
- 公共 API、复杂算法、非显而易见的业务规则（如汇率/手续费计算）**必须**注释。
- Java 公共类/方法用 Javadoc；前端导出函数用 JSDoc 或类型即文档。
- 禁止提交被注释掉的“僵尸代码”和 `console.log` 调试输出（提交前清理，admin 生产构建会移除 console）。
- `TODO` / `FIXME` 必须带责任人或关联事项：`// TODO(xiehaibo): 接入真实风控`。

## 6. 格式化与 Lint

- **以工具为准，不手工对齐**：admin 用 ESLint + Prettier + Stylelint（`pnpm lint`）。client/h5 遵循同样的 Prettier 风格。
- 缩进 2 空格；字符串优先双引号（跟随仓库现有风格）；语句结尾分号保留。
- 提交前本地跑 `pnpm typecheck`（前端）/ `mvn -q compile`（后端），**不得带类型错误提交**。

## 7. 前端状态管理（Pinia）

- 一个业务域一个 store，文件放 `store/modules/`，命名 `useXxxStore`。
- store 只存**跨组件共享**状态；组件局部状态用 `ref`/`reactive`，不要什么都塞 store。
- 异步请求可放 store action，但 action 内只编排 `api/` 调用与状态更新，不直接写 axios。
- 禁止在组件里直接 mutate 其它模块的 store 内部状态，统一通过 action。

## 8. 依赖与导入

- 用路径别名 `@/`（已在各项目 `tsconfig`/`vite.config` 配置），禁止超过两级的 `../../`。
- 不引入功能重复的库（已有 Element Plus / Vant / axios / dayjs 等，先复用再考虑新增）。
- 新增依赖需说明用途并评估体积；移动端（client/h5）尤其控制包体。

## 9. 禁止模拟数据

- **业务代码禁止内置模拟/假数据**：包括硬编码的演示数据、随机造数（faker 等）、写死的列表/详情、前端/后端的 mock 返回。
- 数据一律来自**真实接口或数据库**；前端通过 `api/` 调真实后端，后端通过 `service`/`mapper` 取真实数据。
- 联调/演示需要数据时，走**真实可控的种子数据/测试库**或独立的测试代码，**不得混入生产业务代码路径**。
- 例外仅限**单元测试**内的测试夹具（fixtures），且只存在于 `src/test`。
- 现存的 `server` 演示控制器（`AuthController`/`UserController`/`RouteController` 内的硬编码返回）属于**临时脚手架**，违反本条，须尽快替换为 DB + JWT 的真实实现后删除。

## 10. 构建时质量门禁

- 前端 `pnpm build` 会先执行 `pnpm check`（`typecheck` + `check:lines`），任一不过则构建失败。
- 后端 `mvn package` 在 `validate` 阶段执行行数检查（`exec-maven-plugin` 调 `scripts/check-file-lines.mjs`）。
- **不得为通过门禁而注释/删除检查**；确需豁免历史底座文件，加入 `scripts/linecount-baseline.json` 并注明原因，新代码不得进入基线。
