# 数据库规范

## 1. 命名

- 库/表/字段统一 **小写蛇形** `snake_case`，**禁止**大写、驼峰、中文、保留字。
- 表名用**业务含义单数或统一约定**（团队统一，不混用），可加模块前缀：`trade_order`、`sys_user`、`fund_account`。
- 字段名表意清晰，布尔字段用 `is_xxx`，时间字段用 `xxx_time`，外键用 `xxx_id`。
- 索引命名：主键 `pk_`，唯一 `uk_表_字段`，普通 `idx_表_字段`。

## 2. 金额与数值精度（外汇核心红线）

- 金额、汇率、手续费、点差一律 **`DECIMAL`**，**严禁** `FLOAT`/`DOUBLE`（浮点误差不可接受）。
- 推荐精度：金额 `DECIMAL(20,8)`、汇率 `DECIMAL(20,10)`（按品种精度统一定义并文档化）。
- 大整数主键、订单号用 `BIGINT`，对前端序列化为 `String`（防 JS 精度丢失）。
- 金额相关字段一律 `NOT NULL DEFAULT 0`，禁止用 `NULL` 表示 0。

## 3. 必备字段（每张业务表）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` 主键 | 雪花/自增，全局唯一 |
| `create_time` | `DATETIME` | 创建时间，建议库默认或代码统一填充 |
| `update_time` | `DATETIME` | 更新时间，自动维护 |
| `create_by` / `update_by` | `BIGINT`/`VARCHAR` | 操作人（资金/交易表必填，用于审计） |
| `deleted` | `TINYINT` 默认 0 | **软删除**标志，0 正常 1 删除 |
| `version` | `INT` | 乐观锁（并发更新的资金/订单表必加） |

- **资金相关表禁止物理删除**，一律软删除并保留审计。

## 4. 字符集与表结构

- 统一 `utf8mb4` / `utf8mb4_general_ci`（或团队约定的排序规则），支持 emoji 与多语言。
- 每个字段建议带 `COMMENT` 说明含义与取值（尤其状态枚举）。
- 状态/类型字段用 `TINYINT` + 代码侧枚举映射，并在 `COMMENT` 写明各值含义。
- 单表字段不宜过多（建议 ≤ 30），过宽按职责垂直拆分。

## 5. 索引

- 高频查询条件、外键、唯一约束字段必须建索引；遵循**最左前缀**设计联合索引。
- 避免冗余索引与低区分度单列索引（如性别）；超过 5 个索引需评审。
- 唯一性约束优先用**唯一索引**在库层保证，不只靠应用判重（防并发写入重复）。

## 6. SQL 编写

- **禁止 `SELECT *`**，显式列出字段，按需取数。
- 写操作走参数化（MyBatis `#{}`），禁止拼接（见[安全规范](./03-安全规范.md)）。
- 复杂业务逻辑放应用层，**不写存储过程/触发器/外键级联**（可维护性与可移植性优先）。
- 大批量操作分批；避免大事务长事务锁表。
- 分页用 `LIMIT`/分页插件，禁止把全表拉到内存再分页。

## 7. 事务与一致性（交易特有）

- 资金变动（下单、成交、出入金）必须在**事务**内完成，保证账户余额与流水一致。
- 余额扣减用**乐观锁（version）或行锁**防并发超扣；关键路径做幂等（唯一业务号约束）。
- 每笔资金变动都要有**资金流水记录**（不可变，仅追加），与余额表对账。
- 跨服务/异步场景用消息+对账保证最终一致，禁止靠“补偿脚本手工修数据”。

## 8. 迁移与变更

- 所有表结构变更走**版本化迁移脚本**（Flyway/Liquibase 或团队约定），禁止手工改生产库。
- 迁移脚本只增不改历史；变更需可回滚说明。
- 上线前在预发演练；涉及资金表变更需 DBA + 业务双评审。
