# API 与加密规范

## 1. 统一返回契约（不可破坏）

后端所有接口返回 `Result<T>`，前端以 `code === 0` 判成功：

```jsonc
{ "code": 0, "message": "操作成功", "data": { } }   // 成功
{ "code": 500, "message": "余额不足", "data": null }  // 失败
```

- 成功固定 `code = 0`（`Result.SUCCESS`）；失败用非 0 业务码。
- 失败统一经 `GlobalExceptionHandler` 返回，**不抛裸异常**给前端。
- 新增供 admin 调用的接口，返回结构必须与 `admin/src/api/*` 的 TS 类型对齐。

## 2. RESTful 与版本

- 资源用名词复数，动作用 HTTP 方法：`GET /orders`、`POST /orders`、`PUT /orders/{id}`、`DELETE /orders/{id}`。
- 列表统一分页参数 `currentPage` / `pageSize`，返回 `{ list, total, pageSize, currentPage }`。
- 对外接口加版本前缀（如 `/api/v1/...`）便于演进；后端 `context-path=/api`，前端经 `VITE_API_BASE_URL`(默认 `/api`) + vite proxy 到 `:8080`。
- 接口幂等：`GET/PUT/DELETE` 天然幂等；`POST` 写操作要支持幂等键（防重复下单）。

## 3. 错误码

- 约定分段（示例，团队统一后写入常量类）：`0` 成功；`1xxx` 参数/校验；`2xxx` 鉴权/权限；`4xxx` 业务；`5xxx` 系统。
- 错误码集中在常量/枚举管理，`message` 面向用户友好，细节进日志不进前端。

## 4. 传输安全基线

- 生产**全链路 HTTPS**；登录、支付、出入金等敏感接口禁止明文。
- 鉴权用 `Authorization: Bearer <token>`；token 不放 URL（防日志/Referer 泄露）。

## 5. 请求签名与防重放（开放/敏感接口）

对资金类与对外开放接口，除鉴权外增加**签名**：

- 客户端按约定拼接：业务参数 + `timestamp` + `nonce`（随机串），用密钥做 **HMAC-SHA256** 生成 `sign`，随请求头下发。
- 服务端校验：
  1. `timestamp` 在允许时间窗内（如 ±5 分钟），否则拒绝；
  2. `nonce` 在窗口内唯一（缓存去重），防重放；
  3. 重算 `sign` 与请求一致，防篡改。
- 签名密钥按渠道/应用分配，可轮换；密钥不下发到前端可见处。

## 6. 字段级加解密（敏感数据）

- 敏感字段（证件号、银行卡、手机号等）**落库加密**：对称用 **AES-256-GCM**，密钥由 KMS/环境注入，**不硬编码**。
- 端到端敏感传输（如密码、关键证件）可用 **RSA/SM2** 非对称加密：前端用公钥加密，服务端私钥解密；私钥仅在服务端。
- 密码**只哈希不加密**：`BCrypt`（或 Argon2）加盐存储，不可逆，不可解密。
- 国内合规场景可用国密 **SM2/SM3/SM4** 替代 RSA/SHA/AES，按监管要求选型。
- 加解密逻辑收敛到 `common`/独立工具与 `config`，业务代码只调用，不散落密钥与算法细节。

## 7. 限流与防刷

- 登录、短信、下单等接口做**限频**（按 IP + 账号 + 设备维度），防爆破与刷单。
- 短信/验证码加图形验证或滑块；验证码一次性、有时效。
- 网关/拦截器层做全局限流与黑名单，应用层做业务级幂等与风控。

## 8. 接口文档与联调

- 接口需有文档（OpenAPI/Swagger 或团队约定），包含出入参、错误码、签名规则。
- 字段命名前后端一致（驼峰），由序列化层统一约定，不在业务里手工改名。
- 联调以本规范的 `Result` 契约为准；演示账号见 [CLAUDE.md](../CLAUDE.md)。
